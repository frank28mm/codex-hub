# Retro

Use this skill when the user wants a retrospective for a finished or mostly finished slice of work.

Before using this skill, read `../_shared/gstack_phase2_protocols.md`.

## Goal

Produce a concise retrospective that preserves what happened, what should change, and what to carry forward.

## Workflow

1. Fix the scope:
   - what timebox, version, incident, or project slice is being reviewed
2. Gather the outcome facts:
   - what was delivered
   - what failed or slipped
   - what was learned
3. Separate:
   - what worked well
   - what created friction
   - what remains unresolved
4. Distill the smallest useful changes for next time:
   - process changes
   - tooling changes
   - planning changes
5. Recommend the next move:
   - adopt changes now
   - capture as backlog
   - revisit after more evidence

## Rules

1. Keep blame out; keep evidence in.
2. Separate narrative from verifiable outcomes.
3. Prefer a few durable lessons over a long diary.
4. Make sure each recommended change is actionable.
5. End with what should change next cycle.

## Output Contract

Return in this order:

1. Status
2. Scope reviewed
3. What went well
4. What created friction or remains unclear
5. Recommended changes and next step
